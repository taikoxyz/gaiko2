# TDX-Gaiko2 Image Profile Design

## Goal

Keep the TDX image profile inside the `gaiko2` repository as a deployment
profile for the `tdxgeth` provider mode, not as an independent product.

## Design

The `tdx/` directory owns the image-local service layout and manifest export
for a measured VM image that runs:

- `tdxs`
- `taiko-geth`
- `taiko-client`
- `gaiko2 server` in `tdxgeth` mode

The VM image must bake these binaries, systemd units, and startup scripts into
the measured image. Runtime container pulls, host-mounted startup scripts, or
floating tags are outside the trust boundary and must not be used for accepted
provider images.

## Manifest

The manifest is a release attachment for the gaiko2 TDX image. It records:

- pinned source commits or versions
- binary sha256 values
- systemd unit and startup script sha256 values
- local service endpoints
- final TDX image identity or measurement values used by registration tooling

Generated release manifests should be uploaded with the TDX image artifact. The
example manifest in the repo documents shape only; production manifests are
generated by the image build pipeline.

## Runtime Boundary

`taiko-geth` exposes JSON-RPC only on loopback inside the VM. `gaiko2` verifies
request headers against that local RPC before signing the canonical raiko2
remote prover input hash. `raiko2` talks only to the `gaiko2` remote prover HTTP
endpoint and does not depend on the internal image profile layout.
